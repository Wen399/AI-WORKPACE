# AI Workspace Workflow

> 本文件定义 AI 与开发者之间的固定协作流程。
> 所有 AI 在进入本工作区后，都必须优先遵守本流程。

---

# 工作原则

本工作区遵循：

**一次只解决一个问题。**

不要主动扩展需求。
不要自行增加功能。
不要为了"更完美"而偏离当前目标。

如果发现更好的方案：

先说明原因。

等待确认。

得到确认后再执行。

---

# 每次开发流程

## Step 1：需求确认

开发开始前：

AI 必须确认本次目标。

不要同时修改多个模块。

不要顺便优化其它页面。

只完成本次需求。

---

## Step 2：开发

根据需求：

输出：

- HTML
- JS
- CSS

保持当前项目风格。

尽量在现有版本基础上修改。

不要重写整个项目。

---

## Step 3：开发完成后（必须执行）

每次开发结束时，

AI 必须同时输出：

### ① HTML

本次完整可运行版本。

---

### ② CHANGELOG

说明：

今天修改了哪些内容。

格式：

```markdown
## YYYY-MM-DD

### 新增

-

### 优化

-

### 修复

-
```

如果没有：

写：

```text
无
```

---

### ③ Commit Message

输出一句适合作为 Git Commit 的描述。

例如：

```
优化供应商交付率计算逻辑
```

不要超过一句。

---

### ④ README

判断：

README 是否需要更新。

如果：

项目介绍没有变化。

直接输出：

```
README：无需更新
```

否则：

输出需要修改的内容。

---

### ⑤ SKILL

判断：

业务逻辑是否发生变化。

如果：

没有新的业务规则。

输出：

```
SKILL：无需更新
```

如果：

增加：

新的公式

新的计算逻辑

新的业务规则

新的数据来源

新的限制条件

则输出：

```
SKILL 需要更新：

……
```

---

# 开发者负责

开发者只负责：

① 下载 HTML

↓

② 放入：

AI-workspace

对应项目目录

↓

③ 更新：

CHANGELOG

（复制 AI 提供内容）

↓

④ 更新：

README

（如果需要）

↓

⑤ 更新：

SKILL

（如果需要）

↓

⑥ Commit

↓

⑦ Sync GitHub

---

# AI 不应该做的事情

不要：

- 擅自增加需求
- 擅自修改业务逻辑
- 擅自重构项目
- 擅自改变 UI 风格
- 擅自删除已有功能

如果认为有更好的方案：

先说明。

等待确认。

确认后执行。

---

# 沟通原则

如果开发者说：

> 今天结束。

AI 默认输出：

① HTML

② CHANGELOG

③ Commit Message

④ README 是否更新

⑤ SKILL 是否更新

无需开发者再次提醒。

---

# 项目原则

Workspace 是长期资产。

每一次 Commit，

都代表：

项目成长过程中的一个稳定节点。

AI 的职责：

帮助开发者持续沉淀项目，

而不是仅完成一次回答。
