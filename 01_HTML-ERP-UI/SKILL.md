# 01_HTML-ERP-UI Skill

## 适用场景
当用户要求生成、修改、修复任何业务型 HTML 网页、数据看板、供应链系统、发货资料助手、AI 测试页时，优先调用本 Skill。

## 用户核心设计习惯
- 页面风格偏 ERP、经营分析、办公系统，不要花哨、不要游戏化。
- 逻辑稳定第一，性能第二，UI 美观第三。
- 所有页面必须清晰、干净、留白适中、信息密度合理。
- 不喜欢凌乱的按钮、过窄的卡片、错位的表格、溢出的文本。
- 不要为了好看牺牲数据逻辑。
- 不要随意新增 Sheet、字段、复杂配置，除非用户明确要求。
- 尽量减少用户点击次数，常用信息默认展示。
- 可离线运行的单文件 HTML 优先。

## 页面整体规范
- 顶部：项目标题、版本号、简短说明。
- 数据上传区：集中放在一个清晰区域，上传后给出状态反馈。
- 筛选区：放在页面左侧或顶部固定区域，保持一致。
- 主内容区：卡片、图表、表格分区清晰。
- 重要指标优先放顶部卡片。
- 表格必须支持横向滚动，不能压缩到难以阅读。

## 卡片规范
- 顶部 KPI 卡片尽量一排展示。
- 卡片宽度一致，间距一致，居中对齐。
- 卡片标题简短，数值醒目，辅助说明小字展示。
- 单位必须明确，例如：件、万、%、金额。
- 不要出现含义不清的卡片名称。

## 按钮规范
- 按钮文案必须业务化，例如“缺货 SKU”“未清完”“无出运计划”。
- 常用按钮靠前，危险操作弱化。
- 按钮大小统一、圆角统一、间距统一。
- 不要出现一排按钮高低不齐、左右错位。
- 选中态必须明显。

## 切片器规范
- 支持多选、搜索、清除。
- 默认展示全部。
- 切片器变更后，所有相关图表和表格必须联动。
- 快照版本也必须保留切片器交互。
- 不能出现“正常版能筛选，快照版不能筛选”的情况。

## 表格规范
- 业务员列优先放第一列。
- SKU、品名、供应商、团队等关键字段不能被隐藏。
- 品名过长时截断显示，点击或悬浮显示完整名称。
- 数字列右对齐或居中，文本列左对齐或居中，保持一致。
- 表头固定更好，但不能影响性能。
- 避免空白行被识别为“未分团队”“未分组”。

## 图表规范
- 图表服务业务判断，不追求炫酷。
- 条形图用于排名，趋势图用于月份变化，饼图仅用于结构占比。
- 图例、坐标轴、单位必须清楚。
- 点击图表联动时，不允许页面自动乱跳或下滑。

## Tooltip / 小感叹号规范
用于解释业务口径，例如：
- 供给锁定
- 预测高估
- 需求下修
- 缺货风险
- 冗余风险
- 交付满足率
说明必须短、准、业务可理解。

## 快照版本 UI 要求
- 快照不是静态截图，而是可交互 HTML。
- 正常版有什么筛选、按钮、搜索，快照版也必须保留。
- 修改快照问题时，优先修改正常版导出逻辑，而不是只修快照副本。

## 禁止事项
- 禁止随意重构整体页面。
- 禁止为了新增功能破坏旧功能。
- 禁止生成只有局部代码、需要用户自行拼接的版本。
- 禁止忽略格式问题。
- 禁止让表格、按钮、卡片出现明显错位。

## 输出要求
生成网页时必须给出完整 HTML 文件。修改网页时优先在用户提供的版本上最小修改，并说明改了什么、为什么改、是否影响旧逻辑。
